# position

Figure out where the project sits among its alternatives and how to say so honestly. Merges competitor research, comparison framing, and user-language research, scoped to what's doable without paid SEO/scraping tools.

Read `GROWTH.md` → Differentiators and Competitors first. No Firecrawl, no DataForSEO, no B2B dossiers. The research here is: read the alternatives' own pages, read what users say in the wild, synthesize a position you can defend.

## The landscape

Start from the competitors listed in `GROWTH.md`. If that table is thin, build it: identify the dominant paid incumbent, the closest open-source peer, any legacy/simpler options, and the built-in/default the project replaces. For each, capture its model (free/paid/OSS), its genuine strengths, and how this project differs.

**Verify current facts before publishing any comparison.** Pricing and feature sets change; check the competitor's own site/repo rather than trusting memory or `GROWTH.md` alone if it's stale.

## Honest comparison rules

- **Acknowledge their strengths.** A mature incumbent is mature; a good OSS peer is good. Saying so makes the rest of your claims credible.
- **Differentiate on what's true and durable** (from `GROWTH.md`), not on disparagement. For OSS this is often: free + open source (vs a paid incumbent), native / few dependencies (vs heavier stacks), a genuinely unique capability, privacy/local-first.
- **Be clear who each tool is best for.** "If you want the deepest feature set and don't mind paying, [incumbent]. If you want [this project's differentiators], this." Honest routing builds trust and still wins the users who fit.
- **Never lie about a competitor's features.** Communities cross-check; one caught exaggeration sinks the whole pitch.

## Comparison page / section format

When the user wants an "alternative to [incumbent]" or "vs [peer]" piece (for the README, a landing page, or alternativeTo):

1. **One-line TL;DR** of the difference.
2. **At-a-glance table** (honest ✓ / ✗ / partial).
3. **Where this project is the better pick** (its differentiators).
4. **Where the other tool is the better pick** (be genuine).
5. **Bottom line by user type.**

Target the real search intent: "[incumbent] alternative," "free [incumbent] alternative," "open source [category]."

## User-language research (free sources)

To ground copy in how users actually talk:

- **Reddit**: search `site:reddit.com [incumbent] alternative` and browse the project's niche subs for what people praise and complain about. Capture exact phrases.
- **Hacker News**: search past threads on the category for technical objections and wishes.
- **GitHub**: read issues on competing OSS projects to see unmet needs and feature requests.
- **Reviews / comments**: what makes people switch; what makes them distrust a tool (permissions, telemetry, weight, abandonment).

Extract: the words users use for the problem, the triggers that send them looking, the objections, and the unmet needs. Feed these verbatim phrases into `copy` and `social`.

Guardrails: online voices skew toward power users and complainers; weight recent sources; don't over-generalize from a handful of posts.

## Deliverable

A short positioning brief: the one-sentence position, ranked honest differentiators, a comparison table vs the named competitors, "best for" routing by user type, and a list of real user-language phrases to reuse. Flag any competitor fact you couldn't verify so the user confirms before publishing.
