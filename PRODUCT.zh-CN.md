# Summarize — 产品定义

## 一句话

把 URL / 本地音视频 / 文本 / RSS 订阅内容,变成结构清晰的中文 Markdown 摘要;也可以在独立页面仅下载在线视频。本地跑,普通用户可直接使用。

## 用户

一个普通人。会用 Mac,会拷贝 URL,会拖文件,会订阅播客或博客。**不会用终端,不知道什么是 brew、Python、ffmpeg、yt-dlp**。买了 Anthropic 或 OpenAI 兼容 API key 的人。

## 平台

- **仅 macOS**,最低 **macOS 14 (Sonoma)**。
- **仅 Apple Silicon (arm64)**。Intel Mac 不支持,安装器直接拒绝。
- 分发方式:Developer ID 签名 + Apple notarization 的 .dmg,**不上架 App Store**(沙箱与内嵌可执行文件不兼容)。

## 设计原则

1. **零安装感**:双击 .app 就能用。第一屏只问 API Key、Provider、Base URL、Model 和输出目录,其它全部内嵌或自动。
2. **三方依赖必须内嵌**:yt-dlp 单文件二进制和精简 ffmpeg arm64 可执行文件打包进 `Contents/Resources/`。Whisper 模型首次启动自动下载,带进度条。**不存在"用户需要下载组件"这一步**。模型下载是一次性的,UI 上叫"准备模型",不叫"下载依赖"。
3. **yt-dlp 可感知更新**:App 启动时检查 yt-dlp GitHub release。发现新版时弹出 toast 提示需要更新,并保留"更新"按钮;用户点击后后台下载替换,失败时给清楚中文错误和重试入口。设置中允许用户指定外部 yt-dlp 路径,该路径存在且可执行时优先于内嵌版本和 Application Support 中的版本执行。
4. **依赖不做成管理中心**:用户不需要理解 yt-dlp / ffmpeg / Python / 包管理;只在需要更新 yt-dlp 时看到 toast 和"更新"按钮。
5. **失败要可恢复但不暴露技术细节**:网络断、API 限流、源不可解码、订阅抓取失败、下载失败时,当前任务只显示失败状态和"重试 / 取消"操作;错误原因写入日志系统,不直接展示给用户。下载中的任务还必须可取消。
6. **浏览器 cookies 可选授权**:用户可以主动允许 App 使用本机浏览器 cookies,供 yt-dlp 访问用户自己已登录且有权限观看的内容。App 不承诺绕过付费、会员、地区、DRM 或平台风控限制。
7. **本地优先**:转录在本地(WhisperKit)。只有生成摘要 prompt 和最终摘要两步走外部 LLM API。

## 屏幕

共有五个主屏幕;历史不作为侧边栏独立页面。

### 1. 欢迎 (首次启动)

- API Key 输入框。
- Provider 选择:Anthropic 或 OpenAI 兼容。
- Base URL 输入框:Anthropic 默认官方端点,OpenAI 兼容端点允许用户填写任意兼容地址。
- Model 输入框。
- 输出目录(默认 `~/Documents/Summarize`,可改)。
- "测试连接"按钮:保存前用当前 Provider、Base URL、Model 和 API Key 做轻量连通性验证;成功后显示可用状态,失败只显示通用失败文案,详细原因写入日志。
- 一个按钮:"开始使用";API 连通性测试失败时仍允许用户保存配置,但按钮旁必须保留未验证状态。
- 后台静默启动 Whisper 模型下载,进度显示在主页顶栏。
- 首次启动除了 API 配置外,还需让用户选择转录模型档位(默认根据机器内存和芯片自动推荐);用户可在设置中随时切换。

### 2. 摘要任务

- 大输入框:粘贴 URL,或拖文件进来。
- 标题(可选,自动从源推断)。
- "开始"按钮。
- 顶栏:当前任务总进度条 + 当前阶段行。当前阶段行左侧显示阶段名,右侧显示该阶段内部百分比,例如"下载音频 90%"。总进度条按任务实际阶段动态分配权重,不显示该任务不会执行的阶段。
- 当前任务面板下方显示队列列表;同一时间只运行一个摘要、下载或订阅创建的长任务,其余任务按创建顺序排队。
- 右上角有历史图标按钮,进入摘要历史任务页面;历史页面左上角显示返回按钮,返回摘要任务面板。
- 任务运行中显示"取消"按钮;任务失败当下显示"重试 / 取消"按钮。错误原因写入日志系统,不在 UI 展示。
- 如果 Whisper 模型还在准备,主按钮置灰,显示"正在准备本地转录模型 (45%)"。

### 3. 仅下载视频

- 独立页面,不进入转录和摘要流水线。
- 参考 YTDown 的核心交互:粘贴视频 URL、清空输入、识别资源、选择格式、下载、下载另一个 URL。
- 资源识别后展示视频格式、音频格式、分辨率、扩展名、文件大小(如果 yt-dlp 返回)、预计保存路径。
- 下载中显示进度、速度、剩余时间和"取消"按钮。
- 任务失败当下保留原 URL 和已识别资源,显示"重试 / 取消"按钮。错误原因写入日志系统,不在 UI 展示。
- 页面提供"使用浏览器登录状态"选项,可选择 Safari、Chrome、Edge 或 Firefox cookies;默认关闭。
- 页面右上角有独立历史图标按钮,进入仅下载视频或音频的历史任务页面;下载历史页面左上角显示返回按钮,返回仅下载页面。
- 页面必须显示简短版权与个人使用提示。
- 输出文件写入用户设置的输出目录,可在完成后"在 Finder 中显示"。

### 4. 订阅

- 支持 RSS 播客订阅和 RSS/Atom 博客订阅。
- 添加订阅:粘贴 feed URL,App 自动识别类型、标题和最新条目。
- 订阅列表显示标题、类型、上次检查时间和新条目数。
- 条目列表只保留"生成摘要"和"仅下载媒体"两个任务创建按钮;博客条目没有媒体时不显示"仅下载媒体"。
- 用户点击"生成摘要"或"仅下载媒体"后,立即创建对应任务并进入任务面板;订阅页面不显示任务执行状态、不提供重试、不负责取消。
- 播客条目生成摘要任务走下载音频、转录、摘要流程;博客条目生成摘要任务跳过下载与转录,直接进入阅读内容和摘要流程。
- 条目已有成功产出后,标题颜色变化:已生成摘要和已下载音频使用不同颜色;对应按钮不可继续点击。
- 条目已有任一成功产出后,显示"Finder"按钮,一键打开该条目产出目录。
- 订阅检查失败时只影响该订阅,不阻塞其它任务。

### 5. 设置

- 设置默认显示为紧凑状态行,每行左侧显示名称,中间显示当前供应商名、模型名或能力名,右侧显示小绿点表示可用。
- LLM 配置行默认显示当前 Provider 和 Model,例如"Anthropic / claude-…"。点击行或下拉按钮后展开 API Key / Provider / Model / Base URL,可直接修改并保存。
- LLM 配置展开后提供"测试连接"按钮;测试只验证连通性和认证,不生成正式摘要,不保存测试响应正文。
- 输出目录行默认显示当前目录名或路径摘要,点击后展开选择目录。
- 本地转录模型行默认显示当前 Whisper 模型档位和模型名;模型已可用时显示小绿点,不可用时显示"下载"按钮,下载中显示准备进度。展开后允许在 tiny / small / medium / large-v3-turbo 之间切换,显示对应大致体积和适配建议;切换需要重新下载新模型时,UI 给出明确确认。
- 摘要任务的转录语言行默认显示"中文",允许改为"英文"、"自动检测"或其它 Whisper 支持的语言。
- 浏览器登录状态行默认显示当前授权浏览器名;可用时显示小绿点,不可用或未授权时显示"授权"按钮,点击后展开 Safari、Chrome、Edge 或 Firefox 选择。
- yt-dlp 行默认显示"视频下载组件"或当前可用版本;可用时显示小绿点。如果检查到新版,同时显示"更新"按钮;更新失败时保留"更新"按钮和简短错误。该行允许用户填写本机外部 yt-dlp 可执行文件路径,留空时使用内嵌或自动更新版本。
- 摘要风格行默认显示"双步定制 prompt(默认)";展开后允许切换为"单步固定 prompt(更快、更省 token)"。当前版本两种模式都必须可用,默认行为不变。
- 不显示 ffmpeg、Python 或包管理信息。

## 历史子页面

历史页面只从对应功能页的历史图标进入,不出现在主侧边栏。

### 摘要历史任务

- 列表:标题 / 来源 / 时间 / 状态。
- 只显示已成功完成并产生输出目录的摘要任务,不显示失败任务或取消任务。
- 不提供 Markdown 预览,不提供复制摘要。
- "在 Finder 中显示" / "删除"。
- 用户通过 Finder 打开对应产出目录,自行查看 `.summary.md`、`.transcript.txt` 和 `.summary-prompt.md` 等文件。
- 左上角显示返回按钮,返回摘要任务面板。

### 仅下载历史任务

- 列表:标题 / 平台或来源 / 格式 / 时间 / 状态。
- 只展示仅下载页面创建且已成功完成的视频或音频下载任务,不混入摘要任务,不显示失败任务或取消任务。
- 不显示错误信息;仅下载任务的失败详情统一写入日志系统。
- "在 Finder 中显示" / "删除"。
- 左上角显示返回按钮,返回仅下载页面。

## 输入支持

- 任何 yt-dlp 支持的 URL(YouTube / 播客 / Bilibili / 等)。仅 YouTube 在当前版本作为重点测试和保证体验的源;其它平台属于"可工作但不保证"。
- 独立视频下载页支持 yt-dlp 可识别的视频 URL,默认下载视频文件,不生成摘要。
- RSS 播客 feed。
- RSS/Atom 博客 feed。
- 本地 .mp3 / .m4a / .wav / .mp4 / .mov / .aac。
- 本地 .txt / .md / .srt(跳过转录,直接送 LLM)。

## 摘要流水线

```
URL  → yt-dlp (内嵌) → 英文字幕 / 中文字幕 / 音频-only / 带音轨视频兜底
RSS 播客条目 → enclosure/media URL → yt-dlp 或 URLSession → 英文字幕 / 中文字幕 / 音频-only / 带音轨视频兜底
本地音视频 ───────────────────────────┐
                                     ↓
                            字幕命中 → 字幕文本
                                     ↓
                            无字幕时才解码音视频
                                     ↓
                            ffmpeg 解码/转码 → 16kHz mono WAV
                                     ↓
                            WhisperKit (本地) → 转录文本
RSS/Atom 博客条目 / 本地文本 ───────────┘
                                     ↓
                            LLM 第一步:阅读内容,产出"根据内容定制化风格的摘要 Prompt"
                                     ↓
                            LLM 第二步:基于摘要 Prompt 生成 Markdown 摘要
                                     ↓
                            写 <输出目录>/<标题>-<时间戳>.summary.md
                                            + .transcript.txt
                                            + .summary-prompt.md
```

URL 和 RSS 播客条目用于摘要时,yt-dlp 获取内容的优先级固定为:英文字幕、中文字幕、音频-only、带音轨视频兜底。字幕命中时直接把字幕清洗为文本进入"阅读内容",不下载音频或视频,也不运行本地转录。没有可用字幕时优先下载音频-only 格式;如果站点没有提供音频-only 格式,才允许回退到包含音轨的合并视频格式并用 ffmpeg 提取音轨。摘要任务不应下载 video-only 格式。

WhisperKit 调用必须显式传入设置中的转录语言;不允许依赖模型自动检测语言。原因是 Whisper 类模型在前几秒采样上经常误判语言,导致整段或前半段被转录到错误语言,这是已知风险。

## 任务队列

App 使用单一串行任务队列管理所有耗时任务,包括摘要任务、仅下载任务、订阅条目创建的摘要任务和订阅条目创建的媒体下载任务。同一时间只允许一个任务处于运行中状态;后续任务进入等待队列,显示标题、来源、创建时间和等待状态。

用户从订阅页连续点击多条内容时,App 只创建多个排队任务并跳转到任务面板,不并发启动 Whisper、yt-dlp 或 LLM 请求。当前运行任务完成、失败或取消后,队列自动启动下一个等待任务。

等待中的任务允许取消;运行中的任务允许取消;失败任务保留在当前任务区域并提供"重试 / 取消"。重试时任务回到队列头部,除非当前已有运行任务,此时排在当前运行任务之后。

## 摘要任务阶段与进度权重

进度条不是简单的阶段数量平均值,而是按耗时预期分配的权重条。每个任务创建后先根据输入类型生成阶段计划;不存在的阶段权重必须从总进度中移除,并按该任务实际阶段重新归一化到 100%。

运行中 UI 必须同时表达两个数值:总进度和当前阶段进度。总进度用于进度条填充,按"已完成阶段权重 + 当前阶段权重 × 当前阶段百分比"计算;当前阶段行显示"阶段名"和"当前阶段百分比",不显示总进度百分比。

```
--------------------------------
下载音频                    90%
--------------------------------
```

当阶段无法提供内部进度时,当前阶段百分比显示为不确定状态,文案可使用"处理中"而不是伪造百分比。阶段完成后切换到下一阶段并从 0% 开始显示下一阶段进度。

下表是默认权重,适用于中等长度(20–90 分钟)内容。对显著偏离这个区间的输入(超长播客、极短视频、纯文本),Pipeline 应在创建任务时根据探测到的内容时长动态调整权重——例如三小时播客中转录阶段实际占比可达 70–80%,下载阶段不到 5%。具体调整策略由实现决定,但默认权重不得直接套用到任意长度。

## 日志系统

App 必须内置日志系统,用于用户上报、失败排查和崩溃调查。日志默认写入 Application Support 下的 `Logs/` 目录;每个任务还在自己的输出目录下保存任务级日志引用或副本。

日志不能靠在每个按钮或页面里散落手写 `print` / `console.log`。实现上必须提供统一日志服务,例如 `AppLogger`,各 Store、Service 和任务流水线通过结构化事件调用它。

日志至少分为三类:

- `app.log`:记录 App 启动、退出、设置变更、Provider 切换、模型准备、yt-dlp 更新检查、订阅刷新、页面关键操作。
- `tasks/<task-id>.log`:记录单个任务的阶段开始、阶段结束、进度、取消、重试、输出路径、耗时和使用的非敏感配置摘要。
- `errors/<task-id>.log`:记录任务失败、下载失败、转录失败、LLM 请求失败、cookies 授权失败、yt-dlp 更新失败等错误详情。

关键操作应在状态边界记录,而不是在 UI View 内逐行打印。示例包括:任务创建、任务取消、任务重试、阶段开始、阶段完成、阶段失败、下载完成、转录完成、LLM 请求开始、LLM 请求失败、设置保存、浏览器授权结果、yt-dlp 更新结果、订阅刷新结果。

日志格式必须结构化,推荐 JSON Lines 或 Apple `Logger` 加文件落盘。每条记录至少包含时间、事件名、级别、模块、任务 id(如果有)、阶段(如果有)、非敏感元数据和错误分类。

SwiftUI View 只触发用户意图,日志记录应主要发生在 ViewModel / Store / Service / Pipeline 层,确保同一操作无论来自按钮、快捷键、订阅触发还是重试入口,都写出一致事件。

崩溃调查需要能生成"诊断日志包"。诊断日志包包含最近日志、任务元数据、App 版本、macOS 版本、芯片架构、yt-dlp 可用版本、Whisper 模型状态和崩溃上下文。用户必须手动触发导出;App 不自动上传日志。

日志必须脱敏:不记录 API Key、完整 Authorization header、浏览器 cookies 原文、cookie 文件路径、完整转录文本、完整摘要正文、完整 prompt、用户本地完整文件内容。URL 默认记录域名和哈希,必要时记录完整 URL 也必须经过用户显式同意。

UI 不直接展示技术错误详情,只显示通用失败状态和可操作按钮。详细错误只进入日志;历史页面不显示失败任务,也不展示错误日志内容。

## 仅下载视频流水线

```
视频 URL → yt-dlp 识别资源 → 用户选择格式 → yt-dlp 下载视频文件
                                      ↓
                         写 <输出目录>/<标题>-<格式>.<扩展名>
```

如果用户开启"使用浏览器登录状态",识别资源和下载两个阶段都必须使用同一个浏览器 cookies 来源,避免识别到的格式与实际下载权限不一致。

cookies 在 YouTube 上的行为不稳定:同一个 URL 不带 cookies 时返回的格式列表可能比带 cookies 时更完整,带 cookies 时也可能因账号状态或风控被进一步限制。识别失败时,UI 必须提示用户"可尝试关闭浏览器登录状态重试",并把该操作做成一键。

## LLM 兼容性

- Provider 支持 Anthropic 官方 Messages API。
- Provider 支持 OpenAI SDK 兼容端点,使用 Chat Completions 兼容请求格式。
- Base URL 必须可配置,适配自托管网关、代理服务和第三方兼容供应商。
- API Key 存 Keychain。
- 摘要生成默认走两次 LLM 调用:第一次生成定制化摘要 prompt,第二次按该 prompt 总结。设置中提供"单步固定 prompt"模式,用一次调用完成摘要,体积更小、token 成本更低,作为对成本敏感用户的选项;当前版本默认仍为双步模式。
- 第一步输出的摘要 prompt 必须保存为 `.summary-prompt.md`,便于后续排查和优化。单步模式下,该文件保存内置默认 prompt,以保持产出目录结构一致。

## 转录模型档位

App 在首次启动时根据机器内存和芯片型号给出默认推荐;用户可在设置中切换。
当前版本支持的档位:

| 档位 | 模型 | 体积 | 推荐场景 |
|---|---|---:|---|
| 快速 | tiny | ~75MB | 极短内容、调试、低内存机器(M1 8GB)兜底 |
| 平衡 | small | ~250MB | 多数 8GB 内存机器、内容时长普通、对延迟敏感 |
| 较好 | medium | ~770MB | 16GB 内存以上、对准确度有要求 |
| 高质量(默认推荐) | large-v3-v20240930_626MB(turbo) | ~626MB | 16GB 内存以上、追求最高准确度;同 medium 体积但准确度更接近 large |

默认推荐规则:M1 8GB 默认"平衡(small)";M1 Pro/M2 及以上、16GB 内存或更高,默认"高质量(large-v3-turbo)"。用户可随时切换,切到未下载的档位需要再次下载;切换不影响已经完成的历史任务。

模型档位、模型名、运行时内存峰值、转录耗时和分段进度必须写入任务日志,以支持后续诊断和性能回归判断。

## 关于依赖(实现细节,不暴露给用户)

| 依赖 | 处理方式 |
|---|---|
| yt-dlp | **bundle 内嵌** `yt-dlp_macos` arm64 单文件二进制(~50–70MB)。启动时检查 GitHub release;发现新版弹 toast,用户点击"更新"后新版写入 Application Support,并优先于 bundle 内版本执行。下载得到的二进制必须 ad-hoc 重新签名并清除 quarantine 属性,否则在 Gatekeeper 检查下无法运行。用户开启浏览器 cookies 时,通过 yt-dlp 的浏览器 cookies 能力读取用户选择的浏览器登录状态。优先级:用户指定的外部路径 > Application Support 中的更新版本 > bundle 内版本 |
| ffmpeg | **bundle 内嵌**精简 arm64 构建,只用于音视频解码、抽取音轨和转码到 16kHz mono WAV。摘要任务中的 yt-dlp 先查找英文字幕和中文字幕;无字幕时优先下载音频-only 格式,站点未提供音频-only 时才回退包含音轨的合并视频格式。摘要任务不下载 video-only 格式;仅下载视频页面允许用户选择 yt-dlp 返回的视频格式 |
| Python | **不用**。yt-dlp_macos 是 PyInstaller 自包含构建,整个 app 业务代码是纯 Swift |
| Whisper 模型 | 首次启动后台自动下载用户选定档位的模型(默认根据机器配置推荐),约 75MB 至 770MB,一次性。已下载模型保留在 Application Support 下;切换档位时按需下载新模型 |
| Whisper 推理 | WhisperKit(CoreML,Apple Neural Engine);large 档位运行时内存峰值约 1.2–1.8GB,small/tiny 档位显著更低 |
| LLM | URLSession 直连 Anthropic Messages 或 OpenAI 兼容端点。API Key 存 Keychain |

LLM 连通性测试必须走统一服务层并写入日志。Anthropic 使用一次极短 Messages 请求验证认证、Base URL 和模型名;OpenAI 兼容端点优先使用 `/models` 或等价轻量请求验证连通性,必要时退回 1-token Chat Completions 请求。测试失败不展示原始 HTTP 响应、Authorization 信息或供应商返回的完整错误体。

- 不上架 Mac App Store。
- 不做用户可见的 ffmpeg / Python / 包管理状态展示。
- 不做封面编辑、字幕编辑或复杂媒体库管理。
- 不做云同步。
- 不做多用户 / 账号。

## 已知风险与不承诺事项

以下风险来自 yt-dlp 与 WhisperKit/whisper 社区已确认的问题,产品设计需要正面承认它们而不是假装它们不存在。

**YouTube 风控**:yt-dlp 在 YouTube 上经常返回 "Sign in to confirm you're not a bot" 错误,即使带了 cookies、即使是 nightly 版本。YouTube 同时在做 IP 信誉评分、PO Token、SABR streaming 和 nsig 函数验证,这些机制每隔几周变化。本产品不承诺任意 YouTube URL 都能稳定下载;只承诺:在 yt-dlp 当前可用能力范围内尽力工作,识别失败时给出清晰失败状态、保留原 URL、提供更新 yt-dlp 和切换 cookies 的入口。

**浏览器 cookies 的现实限制**:在 Sonoma 及更高版本上,Safari 的 cookies 文件被 TCC 保护,yt-dlp 默认无法读取,需要用户授予 App "完全磁盘访问"权限;Chrome/Edge/Brave 的 cookies 被 Keychain 加密,读取时可能触发系统钥匙串提示;Firefox 的 cookies 是明文 sqlite,读取最稳定但用户基数小。带 cookies 反而会让 YouTube 返回受限的格式列表,这是上游问题,本产品无法解决。授权 UI 需要对每个浏览器标注大致的预期和必要的系统权限。

**多语言混合内容**:Whisper 对中英混杂、方言、不同语速变化的内容容易在前几秒判错语言,导致整段被错误转录。本产品的对策是默认让用户在设置中指定主语言,而不是依赖模型自动检测;长内容里如果出现局部语言切换,转录结果可能仍然有偏差,这是模型能力本身的边界。

**双步 LLM 的额外成本**:默认的双步 prompt 设计会带来约两倍的 LLM 调用 token 成本和大致两倍的端到端摘要延迟,但产出的摘要风格能更好地适配内容类型。对成本和速度更敏感的用户可在设置中切换到单步固定 prompt 模式。当前版本不强制要求双步带来明显质量提升的客观证据,但保留双步作为默认是基于"内容多样性大、固定模板难以覆盖"的产品判断;后续可根据用户数据调整默认值。

**yt-dlp 自动更新的边界**:Apple notarization 通过的 App 在运行时执行后下载的二进制时需要绕过 quarantine 并做 ad-hoc 签名,否则会被 Gatekeeper 拦截。该流程实现复杂,失败时必须回落到 bundle 内版本或弹 toast 提示用户改用整 App 升级;不允许把用户卡在"yt-dlp 已下载但无法启动"状态。

**Apple Silicon 内存限制**:large-v3-turbo 模型推理峰值在 M1 8GB 上会触发交换,导致整机卡顿。模型档位选择和默认推荐机制的设计直接服务于规避这一点。设置中切换到 large 档位时,如果检测到机器内存不足,给出明确提示。

**B 站等非 YouTube 平台**:B 站没有官方 RSS,需要依赖第三方桥(RSSHub 等);cookies 和接口限制比 YouTube 更严格,yt-dlp 支持也不稳定。本产品把 B 站和其它 yt-dlp 支持的源标记为"可工作但不保证",当前版本只对 YouTube 做体验保证。

**版权与分发风险**:仅下载视频页面在多数司法管辖区属于法律灰色地带,过去发生过类似工具被 Apple 吊销 Developer ID 的案例。本产品的对策是把仅下载视频做成独立页面,UI 显式说明仅供个人有权访问内容的存档使用,默认关闭 cookies,不内置任何专门用于绕过限制的策略或参数。

## 成功标准

一个完全不懂技术的人,从 GitHub Release 下载 .dmg,拖进 Applications,双击打开,粘贴 API Key,选择 Provider、Base URL、Model 和输出目录,就能开始使用。中间不需要看任何说明、不需要装任何东西、不需要回答任何技术问题。

首次启动时,App 必须自动准备本地转录模型,并用清楚的中文进度说明当前状态。模型准备可能需要几分钟,但用户不需要知道 Whisper、CoreML、模型文件或依赖下载是什么。

模型准备完成后,用户粘贴一个常见公开 YouTube URL,或拖入一个本地音视频 / 文本文件,App 必须能自动完成下载、转录、两步摘要生成和文件保存。任务耗时可以随内容长度、网络和 API 响应变化,但过程必须可见、可取消、可重试、可恢复。

用户进入"仅下载视频"页面时,必须能像普通网页下载工具一样粘贴 URL、识别可下载格式、选择视频文件并保存到输出目录;这个流程不能触发转录或摘要。

用户开启浏览器 cookies 后,YouTube 上需要登录但用户已有权限观看的内容,应能在 yt-dlp 支持范围内用该登录状态识别和下载;如果平台限制仍然阻止下载,App 必须给出清楚错误并允许关闭 cookies 或重试。

用户添加播客或博客 RSS 订阅后,App 必须能检查新条目,并允许用户对单条内容创建"生成摘要"或"仅下载媒体"任务。任务创建后立即进入任务面板;订阅页只显示成功产出标记、禁用已完成动作,并提供 Finder 打开产出目录。

产品成功的核心标准不是固定秒数,而是:非技术用户不需要配置任何组件,就能稳定得到一份结构清晰的中文 Markdown 摘要,或在独立页面完成视频下载。
